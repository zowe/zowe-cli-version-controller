Package: org.zowe.pipelines.generic

[Groovy] Class GenericPipeline

      • org.zowe.pipelines.generic.GenericPipeline

      • Extends the functionality available in the Pipeline class. This class adds methods for building and testing your application.

        Required Plugins:
        The following plugins are required:
        • All plugins listed at Pipeline
        • Credentials Binding
        • JUnit
        • HTML Publisher
        • Cobertura
        Example:
         @Library('fill this out according to your setup') import org.zowe.pipelines.generic.GenericPipeline
 node('pipeline-node') {
     GenericPipeline pipeline = new GenericPipeline(this)

     // Set your config up before calling setup
     pipeline.admins.add("userid1", "userid2", "userid3")

     // Define some protected branches
     pipeline.protectedBranches.addMap([
         [name: "master"],
         [name: "beta"],
         [name: "rc"]
     ])

     // Define the git configuration
     pipeline.gitConfig = [
         email: 'git-user-email@example.com',
         credentialsId: 'git-user-credentials-id'
     ]

     // MUST BE CALLED FIRST
     pipeline.setupGeneric()

     pipeline.buildGeneric()  //////////////////////////////////////////////////
     pipeline.testGeneric()   // Provide required parameters in your pipeline //
     pipeline.deployGeneric() //////////////////////////////////////////////////

     // MUST BE CALLED LAST
     pipeline.endGeneric()
 }

          • void buildGeneric(Map arguments = [:])

            Creates a stage that will build a generic package.

            Calling this function will add the following stage to your Jenkins pipeline. Arguments passed to this function will map to the BuildStageArguments class. The BuildStageArguments.operation will be executed after all checks are complete. This must be provided or a NullPointerException will be encountered.

            Stages:
            This method adds the following stage to your build:
            Build: BuildStageArguments.name
            Runs the build of your application. The build stage also ignores any BuildStageArguments.resultThreshold provided and only runs on ResultEnum#SUCCESS#SUCCESS.

            Exceptions:

            The following exceptions can be thrown by the build stage:

            BuildStageException
            When arguments.stage is provided. This is an invalid argument field for the operation.
            When called more than once in your pipeline. Only one build may be present in a pipeline.
            NullPointerException
            When arguments.operation is not provided.

            Note:
            This method was intended to be called build but had to be named buildGeneric due to the issues described in Pipeline.
            Parameters:
            arguments - A map of arguments to be applied to the BuildStageArguments used to define the stage.

          • void deployGeneric(Map deployArguments, Map versionArguments = [:])

            Creates a stage that will execute a version bump and then deploy. test

            Stages:
            This method can add 2 stages to the build:
            Versioning
            This stage is responsible for bumping the version of your application source. It will only be added if versionArguments is a non-empty map.
            Deploy
            This stage is responsible for deploying your application source. It will always execute after Versioning (if present).
            Conditions:

            Both stages will adhere to the following conditions:

            • The stage will only execute if the current build result is ResultEnum#SUCCESS#SUCCESS or higher.
            • The stage will only execute if the current branch is protected.

            Exceptions:

            Both the Version stage and the Deploy stage will throw the following exceptions:

            DeployStageException
            When stage is provided as an argument. This is an invalid parameter for both stages
            When a test stage has not executed. This prevents untested code from being deployed
            NullPointerException
            When an operation is not provided for the stage.

            Note:
            This method was intended to be called deploy but had to be named deployGeneric due to the issues described in Pipeline.
            Parameters:
            deployArguments - The arguments for the deploy step. deployArguments.operation must be provided.
            versionArguments - The arguments for the versioning step. If provided, then versionArguments.operation must be provided.

          • void endGeneric(Map options = [:])

            Signal that no more stages will be added and begin pipeline execution.

            This method wraps the entire pipeline execution in a withCredentials block. The credentials that are loaded represent the credentials stored in gitConfig. This will ensure that any pushes/commits don't expose the credentials in plaintext console output.

            Parameters:
            options - Options to send to Pipeline.endBase

          • void setupGeneric(GenericSetupArguments timeouts)

            Calls Pipeline.setupBase to setup the build.

            Stages:
            This method adds 2 stages to the build:
            Configure Git
            This step configures the git environment for commits and pushes. The gitConfig provided will be injected into the git remote url and the head will point to the proper remote branch. The username and email settings will also be set. If the current build is for a pull request, this step will be skipped.
            Check for CI Skip
            Checks that the build commit doesn't contain the CI Skip indicator. If the pipeline finds the skip commit, all remaining steps (except those explicitly set to ignore this condition) will also be skipped. The build will also be marked as not built in this scenario.
            Note:
            This method was intended to be called setup but had to be named setupGeneric due to the issues described in Pipeline.

          • void testGeneric(Map arguments = [:])

            Creates a stage that will execute tests on your application.

            Arguments passed to this function will map to the TestStageArguments class.

            Stages:
            This method adds the following stage to the build:
            Test: TestStageArguments.name

            Runs one of your application tests. If the test operation throws an error, that error is ignored and will be assumed to be caught in the junit processing. Some test functions may exit with a non-zero return code on a test failure but may still capture junit output. In this scenario, it is assumed that the junit report is either missing or contains failing tests. In the case that it is missing, the build will fail on this report and relevant exceptions are printed. If the junit report contains failing tests, the build will be marked as unstable and a report of failing tests can be viewed.

            The following reports can be captured:

            Test Results HTML Report (REQUIRED)
            This is an html report that contains the result of the build. The report must be defined to the method in the TestStageArguments.testResults variable.
            Code Coverage HTML Report
            This is an HTML report generated from code coverage output from your build. The report can be omitted by omitting TestStageArguments.coverageResults
            JUnit Report (REQUIRED)
            This report feeds Jenkins the data about the current test run. It can be used to mark a build as failed or unstable. The report location must be present in TestStageArguments.junitOutput
            Cobertura Report
            This report feeds Jenkins the data about the coverage results for the current test run. If no Cobertura options are passed, then no coverage data will be collected. For more information, see TestStageArguments.cobertura

            The test stage will execute by default if the current build result is greater than or equal to ResultEnum#UNSTABLE#UNSTABLE. If a different status is passed, that will take precedent.

            After the test is complete, the stage will continue to collect the JUnit Report and the Test Results HTML Report. The stage will fail if either of those are missing. If specified, the Code Coverage HTML Report and the Cobertura Report are then captured. The build will fail if these reports are to be collected and were missing.

            Some tests may also require the use of the gnome-keyring. The stage can be configured to unlock the keyring prior to the tests by passing TestStageArguments.shouldUnlockKeyring as true.

            Exceptions:

            The test stage can throw the following exceptions:

            TestStageException
            When a test stage is created before a call to buildGeneric(Map)
            When TestStageArguments.testResults is missing
            When invalid options are specified for TestStageArguments.testResults
            When TestStageArguments.coverageResults is provided but has an invalid format
            When TestStageArguments.junitOutput is missing.
            When TestStageArguments.operation is missing.

            Note:
            This method was intended to be called test but had to be named testGeneric due to the issues described in Pipeline.

            Parameters:
            arguments - A map of arguments to be applied to the TestStageArguments used to define the stage.
